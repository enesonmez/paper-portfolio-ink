# Project Lessons

Bu doküman, `docs/features` altındaki feature dokümanları oluşturulma sırasına göre okunarak ve **lessons_improvement_analysis.md** rehberliğindeki standartlara göre yeniden yapılandırılarak hazırlandı. Amacı, proje boyunca tekrar eden teknik kararların, doğrulanan mimari kalıpların ve AI agent'lar tarafından hatasız tüketilebilecek yapısal rehberlerin tek noktada toplanmasıdır.

---

## 🌟 Eternal Architectural Rules (Altın Kurallar)

1. **[TS]** TypeScript strict mod aktiftir, `any` kullanımı tamamen yasaktır ve TS 6 `./` path normalizasyonu zorunludur.
2. **[FOUNDATION]** Feature klasörlerinde dikey dilim (vertical slice) mimarisi uygulanır; screen bileşenleri router context'inden bağımsız, saf presentational props ile beslenmeli ve unit test edilmelidir.
3. **[PAGINATION]** SQLite/D1 üzerinde temp B-tree kurulmasını önlemek için listelerde keyset/cursor pagination ve composite index ASC/DESC yön hizalaması zorunludur.
4. **[AUTHZ]** Yetkilendirme dinamik claim-based override (`authorization_state.revision`) modeline dayanır. Hassas mutasyonlarda okuma (`read`) ve eylem (`action/manage`) yetkileri birlikte aranır.

## 3. Route And Slice Architecture Lessons

- **[GÜNCEL / ACTIVE]** **[FOUNDATION]**: Projede en verimli desen, route dosyalarını ince tutup asıl yükleme, action, shared state ve screen kompozisyonunu `app/features/*` altına taşımak oldu.
- **[GÜNCEL / ACTIVE]** **[FOUNDATION]**: `route -> server -> shared -> screen -> components -> lib` akışı hem dashboard hem public alanlarda tekrarlı olarak başarılı oldu.
- **[GÜNCEL / ACTIVE]** **[FOUNDATION]**: Refaktörlerin en faydalı sonucu route modüllerinin sade kalması ve client/server sınırlarının netleşmesi oldu.
- **[GÜNCEL / ACTIVE]** **[FOUNDATION]**: Feature büyüdükçe ortak string sabitleri, form alan adları, status değerleri ve query param isimleri shared katmana alınmadan kod hızla kırılganlaşıyor.
- **[GÜNCEL / ACTIVE]** **[I18N]**: Path-prefix i18n gereksiniminde en düşük riskli desen, locale çözümünü tek tek link helper'larına bırakmak yerine route ağacını `/:locale/*` etrafında kurmak oldu; böylece loader, redirect ve meta üretimi aynı locale kontratı üzerinden ilerledi.
- **[GÜNCEL / ACTIVE]** **[I18N]**: Tüm uygulamayı yatay kesen runtime davranışları bir feature altında tutulduğunda domain sınırları bulanabiliyor; `i18n` gibi root loader, route contract, DB cache ve UI hook'larını birlikte besleyen yapılar `app/shared/*` altında konumlandırıldığında hem feature slice'lar daha temiz kalıyor hem de bu modüllerin app-wide policy olduğu daha net görünüyor.
- **[GÜNCEL / ACTIVE]** **[AUTH]**: Auth gibi app-wide policy modüllerinde en kritik sınır, feature UI state'i ile auth runtime'ını birbirine bağlamamaktır; `LoginFormState` benzeri kontratlar önce `shared/auth` altında alınıp sonra auth factory, session ve login orchestration kodu aynı yatay katmana taşındığında bağımlılık yönü feature -> shared olarak temiz kalır.
- **[GÜNCEL / ACTIVE]** **[CACHE]**: Runtime'a göre adapter seçen ve birden fazla feature ile worker entrypoint tarafından paylaşılan cache facade'ları da `lib` yerine `shared/cache` altında konumlandırılmalı; böylece cache key/payload politikası ile platforma özel adapterler aynı app-wide altyapı katmanında toplanır. ([app/shared/cache/](../app/shared/cache/))
- **[GÜNCEL / ACTIVE]** **[FOUNDATION]**: `app/features/*` içinde hem route slice'ı hem de domain kontratı tutmak semantik gürültü üretiyor; `posts/projects/users/skills/resources` gibi saf kontrat klasörleri `app/domain/*` altına taşınıp `features` yalnızca gerçek screen/loader/action slice'larını barındırdığında import zinciri ve klasör ağacı daha hızlı okunuyor.
- **[GÜNCEL / ACTIVE]** **[FOUNDATION]**: Slice klasörü zaten bağlam verdiği için `dashboard-posts.server.ts` benzeri tekrarlayan dosya adları gereksiz; `server.ts`, `screen.tsx`, `copy.ts`, `state.ts`, `href.ts`, `feed.ts`, `theme.ts` gibi rollerine göre isimlendirilmiş küçük modüller, büyüyen `shared/constants` dosyalarından daha dayanıklı bir desen veriyor.
- **[GÜNCEL / ACTIVE]** **[FOUNDATION]**: Bu repo file-system route keşfi yerine `app/routes.ts` ile manuel route config kullandığı için yeni route dosyası eklemek tek başına yeterli değil; nested path refaktörlerinde çocuk route kayıtları aynı ağaca explicit eklenmezse layout açılır ama outlet boş kalır.
- **[GÜNCEL / ACTIVE]** **[FOUNDATION]**: Manual route config kullanan projede `app/routes` klasörünü URL stringlerini dosya adına gömerek düz tutmak hızla okunmaz hale geliyor; route entrypoint'lerini `public`, `dashboard`, `auth`, `locale`, `system` gibi sorumluluk klasörlerine ayırmak ve URL sözleşmesini yalnızca `app/routes.ts` içinde tutmak daha sürdürülebilir bir desen veriyor.
- **[GÜNCEL / ACTIVE]** **[I18N]**: `/:locale/*` deseniyle legacy unprefixed URL'leri aynı route ağacında birlikte taşırken, `/blog` gibi path'ler route ranking nedeniyle `:locale` parametresine düşüp splat fallback'i hiç görmeyebilir; bu tip backward-compatibility path'lerinde redirect mantığı yalnızca catch-all route'a bırakılmamalı, locale loader seviyesinde de korunmalıdır.
